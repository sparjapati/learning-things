# learning-things

Personal notes on programming concepts, built up one question at a time. Notes are grouped into category folders; each category folder has its own `images/` subfolder for diagrams.

## DSA

- [dsa/dsa-prep-plan.md](dsa/dsa-prep-plan.md) — a 4-phase, 2–3 month DSA interview prep plan for someone who's covered most topics but lacks confidence applying them: overview of all 4 phases (Phase 1 detailed separately in phase-1-diagnostic.md), pattern-notebook drilling with interleaved practice, timed applied practice on curated lists, mock-heavy consolidation, plus common prep mistakes vs. what actually works.
- [dsa/phase-1-diagnostic-tracker.csv](dsa/phase-1-diagnostic-tracker.csv) — ready-to-use tracking sheet for the Phase 1 diagnostic, pre-filled with all 36 problems (topic, problem, difficulty, time limit) mapped to their scheduled day; open in Excel/Sheets and fill in the remaining columns while practicing.

## General

- [general/passkeys.md](general/passkeys.md) — what problem passkeys solve (eliminating the shared-secret weaknesses of passwords), how WebAuthn/FIDO2 registration and login work, why they resist phishing, and synced vs. device-bound passkeys.
- [general/qr-codes.md](general/qr-codes.md) — how a QR code encodes data as a grid, its finder/alignment/timing patterns, and Reed-Solomon error correction.
- [general/totp.md](general/totp.md) — how TOTP generates rotating 6-digit codes from a shared secret + time step (HOTP + HMAC-SHA1 under the hood), and why it's not phishing-resistant the way passkeys are.
- [general/zip-file-format.md](general/zip-file-format.md) — ZIP compression (LZ77 + Huffman/DEFLATE) and the container format's per-file entries plus central directory.

## Java

- [java/garbage-collection-vs-manual-memory-management.md](java/garbage-collection-vs-manual-memory-management.md) — why Java has no `delete` and relies on a tracing garbage collector (GC roots, reachability) while C++ uses deterministic RAII/smart pointers, the design philosophy behind that split (zero-overhead vs. safety), reference-cycle handling, a decision checklist for GC vs. manual/RAII memory management, and common Java memory-leak patterns (static caches, unremoved listeners/ThreadLocals, non-static inner classes, classloader leaks).
- [java/hashmap.md](java/hashmap.md) — how a HashMap achieves near-O(1) lookup via bucket arrays and a hash function, how collisions are chained (and tree-ified in Java 8+), resizing/load factor, the hashCode/equals contract, and why the default capacity is 16 and always doubles (power-of-two bitmask indexing, and the single-bit resize shortcut doubling enables).
- [java/java-memory-management.md](java/java-memory-management.md) — the JVM's runtime data areas (per-thread stack/PC register vs. shared heap/method area), where local variables and instance/static fields actually live, escape analysis, the heap's generational structure (Eden/Survivor/Old, PermGen vs. Metaspace), and the string pool/interning.
- [java/java-program-execution.md](java/java-program-execution.md) — what happens when you run `java Demo`: JVM startup, the classloader delegation hierarchy and lazy loading, how a ClassLoader actually works (loadClass/findClass/defineClass, custom classloaders, (name, loader) type identity, parent-first vs. child-first), what the classpath actually is (`-cp`, manifest `Class-Path`, directory vs. JAR entries) and the module path alternative, linking (verification/preparation/resolution), static initialization, finding and invoking `main`, the interpreter-then-JIT (C1/C2 tiered compilation) execution engine, and JVM shutdown.

## Spring Boot

- [spring-boot/spring-execution-contexts-and-hooks.md](spring-boot/spring-execution-contexts-and-hooks.md) — hook points beyond a single request's path: bean lifecycle callbacks (@PostConstruct/@PreDestroy, BeanPostProcessor), async/deferred execution (@Async, @Scheduled, message queues), event publishing (ApplicationEventPublisher, @TransactionalEventListener), transaction-boundary hooks, ambient context holders (RequestContextHolder in detail, SecurityContextHolder, MDC), context-scoped beans, a decision checklist for choosing between them, how uncaught exceptions are handled (or silently swallowed) in each one, and the centralized-handling mechanism that plays @ExceptionHandler's role per context (AsyncUncaughtExceptionHandler, ErrorHandler, Kafka's CommonErrorHandler, etc.).
- [spring-boot/sql-injection-protection.md](spring-boot/sql-injection-protection.md) — why Spring Boot has no dedicated anti-SQL-injection feature: protection comes from parameterized queries/prepared statements, which Spring Data JPA and JdbcTemplate generate automatically when used as intended (derived methods, `:name`/`?` placeholders, Criteria API); what a prepared statement actually is (raw JDBC prepare/bind/execute lifecycle, batch reuse, its distinct security vs. performance benefits); plus safe-vs-unsafe pattern tables (native queries, string concatenation, unbindable identifiers like column names).
- [spring-boot/spring-request-lifecycle.md](spring-boot/spring-request-lifecycle.md) — how an HTTP request flows through Tomcat, filters, DispatcherServlet, RequestBodyAdvice, interceptors, the controller, AOP proxies, service/repository layers, and back out via message converters and ResponseBodyAdvice, plus the exception-handling path (HandlerExceptionResolver, @ExceptionHandler resolution order) and supporting beans (HandlerAdapter, WebMvcConfigurer, Validator, ConversionService, ObjectMapper, CORS, async support).

## System Design

- [system-design/cap-theorem.md](system-design/cap-theorem.md) — CAP theorem: consistency vs availability vs partition tolerance, what a network partition actually is, a decision checklist for CP vs AP with worked examples (bank ledger, leader election, shopping cart, like counts), and example systems for each.

- [system-design/rate-limiters.md](system-design/rate-limiters.md) — common rate limiting strategies (fixed window, sliding window log/counter, token bucket, leaky bucket) and Uber's distributed control-plane architecture (local probabilistic dropping instead of per-request Redis lookups).
- [system-design/redis-persistence-rdb-vs-aof.md](system-design/redis-persistence-rdb-vs-aof.md) — Redis persistence: RDB snapshots vs AOF write log, trade-offs and when to use each.
- [system-design/redis-single-threaded.md](system-design/redis-single-threaded.md) — how Redis works, why it's called single-threaded, why `INCR` is atomic, and use cases beyond caching.
