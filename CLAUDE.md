# learning-things

This is a personal learning project for exploring programming concepts.

## Environment Setup (new device/session)

Diagrams in this project depend on the `eraser` MCP server (see the diagram rules under Workflow below). On a fresh device/environment where it isn't yet configured:

1. Add the server: `claude mcp add --transport http eraser https://app.eraser.io/api/mcp`
2. Start a new Claude Code session (newly-added MCP servers only load into a fresh session).
3. Authenticate: run `/mcp` in that session, select `eraser`, and complete the OAuth login to the Eraser.io account that owns this project's diagrams (`parjapatsanjay1999@gmail.com`, team "Sanjay's Team") — diagrams created under a different account/team won't be visible to it.

## Workflow

- When the user asks about a concept, explain it in the conversation with example with some real life analogies, then also write a summary of it to a markdown file.
- Concept files live inside a category folder (e.g. `system-design/`, `spring-boot/`, `java/`, `kotlin/`, `general/`). A concept that doesn't fit any existing specific category goes in `general/` — don't invent a new specific category folder for a single one-off file; ask the user before creating a new specific category if it's unclear which one (existing or new) a concept belongs to.
- Each category folder has its own `images/` subfolder alongside its concept files (e.g. `system-design/images/`), referenced from files in that folder with a relative path (e.g. `images/qr-code-anatomy.png`).
- File naming convention: kebab-case, e.g. `event-loop.md`.
- Keep the root `README.md` continuously up to date as an index, grouped into a section per category: whenever a concept file is created, add a one-line entry for it under its category's section (linking the category-prefixed path, e.g. `system-design/event-loop.md`, with a short description of what it covers). Whenever an existing concept file gains a new section that changes what it covers, update that file's description line in `README.md` to match.
- In the root `README.md`, always keep category sections sorted alphabetically ascending by category name, and within each section keep the entries sorted alphabetically ascending by filename. Re-sort whenever a category or entry is added, moved, or renamed.
- Diagrams or example images that help explain a concept go in that concept file's category `images/` folder. This includes flow/sequence diagrams (e.g. showing steps between services, request/response flow, or topology comparisons like orchestration vs choreography) as well as structural/anatomy diagrams — generate one whenever a visual would make the steps or shape of a concept clearer than prose alone.
  - **All diagrams are created on Eraser.io via the `eraser` MCP, never by any other/manual method.** All of them live inside a **single Eraser file named `learning-things-project`** (fileId `JLgRjFjapzOnrAqixpQO`) — the account's plan has a hard file-count limit, so new diagrams are added *into* this one file (`targetFileId`/`fileId: JLgRjFjapzOnrAqixpQO`), never as a separate new file per concept.
  - **The account's free AI-generation credits are exhausted** — `create_diagram` and `update_diagram` (both AI-based) will fail with an AI-credits error until the plan is upgraded or credits reset. Until then, use **`manually_create_diagram`** (writes hand-written DSL verbatim, no AI credits used) targeting `fileId: JLgRjFjapzOnrAqixpQO`. Write the DSL yourself following the patterns already in that file (grouped nodes `Name [color: ..., icon: ...] { ... }`, connections `A > B: label`, sequence-diagram constructs like `activate`/`alt {} else {}`/`opt {}`). Check periodically whether AI credits/`create_diagram` have come back before defaulting to hand-written DSL.
  - Export each diagram (`export_diagram`, PNG, using its `fileId`/`diagramId`) into the category's `images/` folder, embed the PNG in the concept file, and also link the editable diagram URL (`https://app.eraser.io/workspace/JLgRjFjapzOnrAqixpQO?diagram=<diagramId>&layout=canvas`) right below it so there's always an editable source, not just a static image.
  - **Keep the Eraser diagram in sync whenever the underlying concept/flow changes** — if a concept file's steps, structure, or comparison are added, removed, or reordered, update the existing diagram in place (`update_diagram` if AI credits are available, otherwise re-submit the full DSL via `manually_create_diagram` using the same `diagramId`'s file), then re-export the PNG and overwrite the same file in `images/` so the checked-in image never drifts from what's linked in Eraser or what the prose describes.
  - **Pre-existing manually-created images** (not sourced from Eraser) are being migrated to Eraser-created equivalents one at a time, not in bulk — replace one image (create the diagram in `learning-things-project`, export, overwrite the file, update the embed + editable-link) fully before moving to the next, so each replacement can be checked. Skip migrating an image if it's not actually a diagram (e.g. a rendered example artifact like a sample QR code image, which Eraser can't generate) — leave those as-is.
- Include short illustrative code or pseudocode snippets (in a fenced code block) when they'd make an abstract pattern concrete — e.g. showing what an orchestrator's step-by-step calls look like versus what a choreographed service's event handler looks like. Keep snippets minimal and language-agnostic/pseudocode unless a specific language is more illustrative; they're there to clarify the shape of the pattern, not to be a runnable implementation.
- **When a concept involves choosing between two or more approaches/options (e.g. CP vs AP, SQL vs NoSQL, orchestration vs choreography)**, always answer the question "when designing a system, how do you decide which one applies?" with its own section — a decision checklist/criteria (questions to ask about the system's requirements, not just a description of what each option is), plus named real-world examples for each option. e.g. `choosing-sql-vs-nosql.md`'s "Decision checklist" table, and CAP theorem's "CP vs AP in practice" table (behavior during a partition + example systems like ZooKeeper/etcd vs Cassandra/DynamoDB) in [system-design/cap-theorem.md](system-design/cap-theorem.md).
  - **The decision checklist itself must include a concrete real-life example for each option, per question** — not just a separate "worked examples" list at the end. For every row/question in the checklist, add a real-life scenario illustrating each option's answer (e.g. a "Real-life example (CP)" / "Real-life example (AP)" column per row), so each decision criterion is immediately grounded rather than left abstract. See the "How to decide: CP or AP for your system?" table in [system-design/cap-theorem.md](system-design/cap-theorem.md) for the reference format.
- **Every `interview-prep/` entry must contrast what should *not* be said against a more appropriate answer** — never just describe the good answer alone. Structure each entry as: the prompt/question, a weak/inappropriate response (marked ❌) with why it undersells or backfires, a stronger/more appropriate response (marked ✅) with why it works, and the general principle behind the contrast. See `interview-prep/recruiter-and-interviewer-responses.md`'s entries for the reference format.

## Handling related questions across sessions

Before writing a new file, check if an existing `.md` file anywhere in this directory (its own category folder first, then other category folders and the root) already covers the same or a closely related concept (by filename and by skimming content). If one exists, decide as follows:

- **Small/tightly related addition** (a follow-up, clarification, or sub-topic that naturally belongs with the existing concept, e.g. an example, edge case, or "how do I decide" companion to an already-documented comparison) → add a new section to the existing file instead of creating a new one.
- **Big/standalone new concept** (substantial enough to be useful on its own, would make the existing file too long or unfocused, or someone might want to find it independently) → create a new file, and add a `See also: [existing-concept.md](existing-concept.md)` (or a short "Related" line linking the filename) pointing back to the existing one. Also add a matching backlink in the existing file if it doesn't already reference the new one. Place each "See also" line wherever it's contextually relevant in the file (e.g. right above the section it relates to) — not always at the top by default. Always use standard markdown link syntax (`[text](file.md)`), never `[[wikilink]]` syntax — GitHub's renderer doesn't convert double-bracket wikilinks into links, so they'd show up as literal brackets.

When unsure whether something counts as "small" or "big," lean toward: if the explanation is a few paragraphs or less and stays fully on the same topic, append; if it introduces a genuinely separate concept that stands on its own, create a new file with a cross-reference.
